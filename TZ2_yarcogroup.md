## Общая задача

Построить **корпоративный сайт-витрину компании YARCO GROUP** на домене **yarcogroup.ru**. Компания —
**поставщик медицинских расходных материалов и средств гигиены для медицинских организаций** (клиники,
лаборатории, стационары). Бизнес-цель — представительский сайт с понятной навигацией, единой фирменной
стилистикой и заделом под управляемый каталог, обмен с 1С/CRM и ИИ-поиск в будущем.
> Аудитория/продукт уточнены заказчиком 2026-06-24: это **медицина**, не «бизнес/HoReCa». Структура меню из
> ТЗ #1 сохранена (категории «Средства гигиены» / «Непродовольственные товары»), наполнение — медицинское.

Директория проекта: `~/projects/yarcogroup/`. Архитектурные стандарты, которые демонстрируем (как в эталоне
`erp-prototype-grace`): **изоляция слоёв** (контент/данные ⟂ UI), **семантический экзоскелет GRACE**
(`//#region` + TSDoc-контракты + GREP/STRUCTURE + мини-блок-схемы), **LDD `[IMP:1-10]`** в браузере,
**AX-Tree-операбельность** (role/aria на каждом интерактиве) и **headless-тестирование** (Vitest по ролям +
Playwright E2E).

**Дорожная карта (зафиксирована, реализуется по итерациям):**
- **v1 (этот документ) — «Витрина»:** все страницы и меню/подменю по структуре ТЗ, дизайн-система из брендбука,
  контент-заглушки; раздел «Продукты» с демо-товарами и рабочим клиентским поиском по ним. **Без серверного
  бэкенда и админки.** Данные витрины — статические декларации в коде (stub за `api.ts`-швом).
- **v2 — «Движок управления»:** FastAPI-бэкенд + БД (object-relational ядро) + админка на метадата-формах
  (добавление страниц / товаров / фото), серверный каталог и поиск через SQL VIEW. UI v1 переключается на
  реальный API без переписывания (stub ⇄ fetch seam).
- **v3 — «Интеграции»:** обмен с 1С, взаимодействие с CRM, ИИ-поиск по товарам.

В v1 код пишется так, чтобы v2/v3 НЕ требовали переписывания витрины: весь доступ к данным идёт через единый
шов `src/lib/api.ts` (сейчас отдаёт stub-декларации, позже — `fetch` в FastAPI).

## Бизнес-требования и функциональность

### 1. Конфигурация и контент-декларации (state)
Весь контент витрины v1 — **типизированные декларации в коде** (не БД), за швом `api.ts`:
- `src/content/site.ts` — глобальные данные: название «YARCO GROUP», контакты (почта, телефон), реквизиты
  (минимальные), ссылка на политику конфиденциальности, тексты футера. *(см. Приложение А)*
- `src/content/nav.ts` — **метадата меню** (пункты + подменю), единственный источник истины для шапки и роутинга.
- `src/content/pages/*.ts` — контент каждой страницы (заголовки, абзацы-заглушки, иконки из брендбука).
- `src/content/products.ts` — **демо-товары** (две категории: «Средства гигиены», «Непродовольственные товары»)
  для раздела «Продукты» и клиентского поиска.
Чего НЕ хранит: секретов, ключей. В v2 эти декларации заменяются ответами FastAPI через тот же тип-контракт.

### 2. Интерфейс / структура страниц (точка входа)
SPA на React + TS + Vite, роутинг (react-router). Раскладка строго по ТЗ и брендбуку.

**Шапка (header), на всех страницах:**
1. Слева — **знак компании** (кольцо-«О» из брендбука, тот же SVG что favicon) + текст «YARCO GROUP».
2. Контакты / почта — в верхней строке шапки (utility-bar). *(в ТЗ сказано «вверху слева» дважды — см.
   Допущения; по умолчанию знак слева, контакты в правой части верхней строки.)*
3. Ниже — **строка основного меню** (6 пунктов). При наведении на пункт с подменю — **выпадающее подменю**
   (hover на десктопе; на тач/мобайл — раскрытие по тапу; доступно с клавиатуры — `role="menu"`/`menuitem`).

**Карта информационной архитектуры (меню → маршруты), 1:1 с ТЗ:**
1. **Главная** — `/`
2. **О компании** — `/about` · подменю: Наша миссия `/about/mission`, Направления бизнеса `/about/directions`
3. **Продукты** — `/products` · подменю: Средства гигиены `/products/hygiene`,
   Непродовольственные товары `/products/non-food`
4. **Партнёрам** — `/partners` · подменю: Инвестиции `/partners/investments`, Поставщикам `/partners/suppliers`
5. **Блог** — `/blog`
6. **Контакты** — `/contacts`

**Главная (`/`):** блок текста о компании / возможен баннер / рисунок (контент-заглушка; верстается секциями
в стиле брендбука — eyebrow + тонкий H1·200 + lead). Допустимо «может ничего» — секции опциональны через контент.

**Раздел «Продукты» (`/products`):** в самом верху, **широкой полосой на всю ширину контента** —
**поиск по товарам** (см. п.3). Ниже — две категории (иконки из брендбука: Гигиена / Товары для дома и т.п.)
и грид демо-товаров.

**Футер (footer), на всех страницах:** почта, телефон, ссылка «Политика конфиденциальности», минимальные
реквизиты (назв., ИНН/ОГРН-заглушки), копирайт. Тонкие hairline-разделители (`--hair`).

### 3. Основной рабочий процесс (поиск по товарам, v1 — клиентский)
1. Пользователь открывает `/products` (или подкатегорию).
2. Вверху — полоса поиска (`role="search"`, `<input aria-label="Поиск по товарам">`).
3. Ввод строки → фильтрация демо-товаров клиентски по нормализованному `search_blob`
   (имя + категория + артикул), регистронезависимо. **NB: для кириллицы НЕ использовать `String.toLowerCase`
   как единственную нормализацию в сравнении? — допустимо в JS (`toLocaleLowerCase('ru')`), но нормализовать
   заранее в `search_blob` при сборке демо-данных** (анти-паттерн `SQL LOWER()` для кириллицы — про бэкенд v2).
4. Результаты — грид карточек товара (фото-заглушка, название, категория). Пустой результат — явное сообщение.
5. Поиск логируется LDD: `[IMP:7][search][QUERY] q="..." hits=N`.
В v2 этот же UI-компонент переключается на серверный поиск через `api.searchProducts()` (шов уже есть).

### 4. Декларации данных / схемы
Типы и метадата — в Приложении А (TS-типы меню, страницы, товара; готовы к вставке). Ключевое:
- `NavItem` (метадата меню → рендер шапки и генерация роутов).
- `Product` + `search_blob` (демо-каталог + клиентский поиск).
- `SiteConfig` (контакты/реквизиты/футер).
- `SiteApi` — **stub-swappable seam** (v1: возвращает декларации; v2: `fetch` в FastAPI), зеркало будущих
  Pydantic-схем.

### 5. Логирование (LDD 2.0, браузер)
Browser-LDD как в `src/lib/ldd.ts` эталона: формат `[IMP:n][FUNCTION][BLOCK] message`, `console` каналы по
важности. Критические точки витрины с `[IMP:7-10]`: монтирование роутера, навигация по меню, поиск (запрос+число
результатов), переключение stub⇄api. `[IMP:9-10]` — AI Belief State (ожидание vs факт) в обработчиках.

## Требования к разработке и архитектуре

1. **Семантический экзоскелет (GRACE):** каждый `.ts/.tsx` начинается с `//#region MODULE_CONTRACT
   [DOMAIN(n); CONCEPT(n); TECH(n)]` + TSDoc-контракт (`@file/@brief/@purpose/@scope/@invariants/@rationale/
   @modulemap`), затем `// GREP_SUMMARY:` и `// STRUCTURE:` с мини-блок-схемой (`▶ ◇ ⊕ ⟶ ⚡ →`). Doxygen/TSDoc-стиль,
   НЕ legacy `# START_MODULE_CONTRACT`. Нулевой дрейф относительно `erp-prototype-grace` и брендбука.
2. **Изоляция слоёв:** контент-декларации (`src/content/**`) и шов данных (`src/lib/api.ts`) ⟂ презентация
   (`src/components/**`, `src/pages/**`). Компоненты не знают, stub это или сеть.
3. **Дизайн-токены — единый источник:** `src/styles/tokens.css` = блок `:root` из брендбука (Графит `--ink #222`,
   Лайм `--green #8CC63E`, `--green-dark #5E9C2E`, `--green-soft #BCE08A`, Белый, `--grey #9A9A9A`,
   `--hair #ECECEC`, `--bg-soft #FAFAFA`; Montserrat; трекинги; `--radius:14px`; `--maxw:1120px`).
   **Никаких жирных выделений** (ТЗ): заголовки 200–300, текст 400, подписи 500+caps+трекинг. Без `font-weight:700`.
4. **AX-Tree-операбельность:** каждый интерактив = `role` + accessible name (меню `role="menu/menuitem"`,
   поиск `role="search"`, кнопки `aria-label`). Никаких «голых» `div` для кликов (анти-паттерн).
5. **Stub-swappable seam:** весь доступ к данным — через `SiteApi` в `api.ts`. Переключение на бэкенд v2 —
   только через env `VITE_API_BASE`, без правок компонентов.
6. **Метадата-driven навигация:** шапка и список маршрутов рендерятся из `nav.ts` (а не хардкодом JSX на пункт) —
   перенос паттерна метадата-форм эталона на меню.
7. **Адаптивность:** сетки `g-2/g-3/g-4/g-6` и медиа-брейкпоинты как в брендбуке (≤880px, ≤520px).

## Стратегия тестирования (бизнес-логика витрины в `frontend/tests/` и `*.test.tsx`)

Vitest + jsdom (юнит, по ролям) и Playwright (E2E, только `getByRole`), `conftest`-аналог — Anti-Loop. Проверяют:
1. **Атомарные:** `nav.ts` → корректные маршруты/подменю; `products.ts` → у каждого товара заполнен `search_blob`;
   `ldd()` формат `[IMP:n]`.
2. **Поиск:** запрос «гиг» находит товары гигиены; кириллица регистронезависимо; пустой результат → сообщение.
3. **Навигация (E2E):** по ролям открыть меню → подменю → перейти на `/products/hygiene`; шапка/футер на всех
   страницах; поиск-полоса вверху `/products`.
4. **Конфиг:** `site.ts` отдаёт контакты/реквизиты/политику; футер их рендерит.
5. **Anti-pattern guard:** в DOM нет кликабельных `div` без role; нет `font-weight:700` в собранных стилях.
6. **Тесты не молчат:** каждый тест печатает `[IMP:7-10]` belief-строку (console-capture). Зелёный assert без
   belief-лога = провал (Semantic Trace Verification).

export type ProductCategory = 'hygiene' | 'non-food';
export interface Product {
  id: string;
  name: string;
  category: ProductCategory;
  article?: string;
  imageUrl?: string;        // v1: заглушка
  search_blob: string;      // нормализованный (lower, ru) name+article+category — для клиентского поиска

  brand: 'YARCO GROUP';
  email: string;
  phone: string;
